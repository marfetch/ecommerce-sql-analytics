# Итоговый бизнес-отчет

## Краткое резюме проекта

В проекте **E-commerce SQL Analytics** реализована аналитическая база данных интернет-магазина в PostgreSQL. Модель покрывает клиентов, каналы привлечения, товары, категории, заказы, состав заказов, оплаты, доставки и отзывы.

Проект показывает полный цикл SQL-аналитики: проектирование БД, создание структуры, генерация данных, построение витрин, расчет метрик, сегментация, аналитические запросы и проверки качества данных.

## Что было реализовано

- Реляционная модель данных интернет-магазина.
- Таблицы с первичными и внешними ключами.
- Ограничения `NOT NULL`, `UNIQUE`, `CHECK`.
- Справочники каналов и категорий.
- Синтетические данные для анализа.
- Индексы для аналитических запросов.
- Представления:
  - `v_order_details`;
  - `v_customer_orders`;
  - `v_product_sales`;
  - `v_delivery_performance`.
- PL/pgSQL функции и процедуры.
- Аналитические запросы.
- Проверки качества данных.
- Документация и шаблоны отчетов.

## Ключевые направления анализа

| Направление | Что анализируется |
|---|---|
| Продажи | Выручка, прибыль, средний чек, динамика по месяцам |
| Клиенты | Повторные покупки, RFM, сегменты, города |
| Товары | Топ товаров, категории, ABC-анализ, рейтинги |
| Маркетинг | Каналы привлечения, выручка по каналам, revenue per customer |
| Доставка | Сроки, задержки, статусы, влияние на отзывы |
| Качество данных | Полнота, согласованность и корректность данных |

## Итоговые бизнес-выводы

По итогам выполнения аналитических SQL-запросов были получены следующие выводы:

- какие месяцы, товары и категории формируют основную выручку;
- насколько прибыльны продажи с учетом себестоимости;
- какие клиенты имеют наибольшую ценность;
- какие каналы дают больше выручки на клиента;
- какие товары требуют развития, контроля наличия или пересмотра;
- где возникают проблемы доставки;
- какие ошибки данных могут искажать аналитику.

## Фактическое резюме анализа

По результатам SQL-анализа база содержит 1 753 доставленных и оплаченных заказа. Общая выручка составила 132 061 089,30, валовая прибыль — 50 347 591,59, валовая маржа — 38,12%.

Основная выручка сконцентрирована в категории `Home Appliances`, которая принесла 95 677 403,51 выручки. При этом категория имеет более низкую маржинальность — 32,08%, чем `Accessories`, `Kids` и `Clothing`.

ABC-анализ показал высокую концентрацию выручки: товар `Home Appliance 42` относится к группе `A` и формирует 72,45% общей выручки. Это создает риск зависимости бизнеса от одного ключевого товара.

Среди каналов привлечения наибольшую выручку показал `referral` — 18 653 628,24. Максимальная выручка на клиента наблюдается у `context_ads` — 236 224,49.

По городам лидирует Москва с выручкой 17 179 775,11. Однако максимальная выручка на клиента наблюдается в Воронеже — 227 882,90.

Доля доставок вовремя составляет 39,36%, доля доставок с задержкой — 31,24%, незавершенные доставки — 29,40%. Это делает доставку одной из ключевых зон для дальнейшего анализа.

## Рекомендации по продажам

- Анализировать выручку вместе с валовой прибылью и маржой.
- Отслеживать средний чек по месяцам.
- Проверять периоды падения выручки через категории, каналы и статусы заказов.
- Отдельно учитывать отмены и возвраты.

## Рекомендации по клиентам

- Использовать RFM-сегментацию для разных сценариев коммуникации.
- Разделять one-time, repeat, loyal и VIP клиентов.
- Развивать удержание клиентов с повторными покупками.
- Реактивировать неактивных клиентов отдельными кампаниями.

## Рекомендации по товарам

- Контролировать наличие товаров группы `A`.
- Анализировать низкомаржинальные товары с высокой выручкой.
- Использовать рейтинги как дополнительный сигнал качества ассортимента.
- Проверять товары группы `C` на актуальность и оборачиваемость.

## Рекомендации по маркетинговым каналам

- Сравнивать каналы по клиентам, заказам, выручке и revenue per customer.
- Не делать выводы об окупаемости без расходов на рекламу.
- Добавить таблицу marketing spend для расчета CAC, ROMI и payback.
- Проверять качество аудитории через повторные покупки.

## Рекомендации по доставке

- Отслеживать долю доставок вовремя и с задержкой.
- Анализировать города с высокой средней задержкой.
- Проверять связь задержек с низкими отзывами.
- Добавить причины failed-доставок и возвратов.

## Ограничения проекта

- Данные синтетические и не отражают реальный бизнес.
- Нет расходов на маркетинг, поэтому нельзя рассчитать CAC и ROMI.
- Нет данных о сессиях, просмотрах, корзинах и конверсии.
- Нет полноценной модели возвратов на уровне товарных позиций.
- Нет BI-дашборда и автоматического пайплайна обновления данных.

## Дальнейшее развитие

- Добавить маркетинговые расходы и рассчитать CAC, ROMI, LTV и payback.
- Добавить событийную аналитику: просмотры, добавления в корзину, checkout.
- Реализовать retention cohorts.
- Подключить BI-дашборд.
- Перенести проект в Docker Compose.
- Добавить dbt-модели и тесты качества данных.
- Расширить синтетические данные сезонностью, промо и возвратами.
